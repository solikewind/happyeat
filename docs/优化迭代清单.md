# HappyEat 优化迭代清单（分步推进）

本文档用于**滤出现阶段需要优化的点**，并按优先级排序，便于团队「一步一步」落地。条目尽量与当前仓库实现对应；完成后可在对应条目前打勾或注明 PR/日期。

---

## 一、文档怎么用

1. **只推进「当前阶段」区块**：不必一次做完；每完成一条，更新状态（如 `[x]` 或备注合并请求链接）。
2. **区分「缺陷/风险」与「能力缺口」**：前者建议优先修；后者按营业场景选做。
3. **与前后端对齐**：部分项同时涉及 `happyeat`（API）与 `happyeat-web`（界面）。

## 二、当前能力速览（基线）

| 域 | 已有能力（摘要） |
|----|------------------|
| 认证与权限 | JWT 登录；IAM；RBAC + Casbin；路由见 `app/internal/handler/routes.go` |
| 菜单与规格 | 菜单 CRUD、分类、分类规格模板、规格组/项 |
| 桌台 | 桌台/桌类 CRUD；表结构含 `status`（idle/using/…）、`qr_code` 字段 |
| 订单 | 创建、分页列表、详情、按目标状态更新（经状态机校验） |
| 工作台 | `GET /workbench/orders` 默认筛 `created/paid/preparing` |
| 前端 | 点餐台、订单管理、工作台、桌台平面图、菜单管理等页面 |

## 三、待优化项（先滤出的清单）

以下为**相对当前代码与常见营业系统**的差距或风险点，未按「必须商用」一刀切，便于你们裁剪。

### A. 数据一致性与安全（建议优先）

| ID | 现状 / 风险 | 建议方向 |
|----|-------------|----------|
| A1 | 下单入参含 `unit_price`、`total_amount`，**服务端未按菜单库重算**，存在改价/篡改风险 | 创建订单时按 `menu_id` + 规格解析服务端计价；总额校验与行项目一致 |
| A2 | `CreateOrderItemReq` 无 `menu_id`；`dal/model/order` 创建明细时**未写入 `menu_id`**（库表支持可选 `menu_id`） | API 与落库携带 `menu_id`，便于对账、报表与防伪造菜名 |
| A3 | API 注释写「堂食必填 `table_id`」，但 `CreateOrderLogic` **未强制校验** `dine_in` 必须带桌 | 堂食无桌拒绝创建；外带禁止绑桌（或明确规则） |
| A4 | 订单号 `genOrderNo` 使用毫秒 + **短随机数**，高并发下**碰撞概率**不可忽视 | 雪花/分段序号/DB 唯一约束重试等策略 |

### B. 桌台与订单业务闭环

| ID | 现状 / 风险 | 建议方向 |
|----|-------------|----------|
| B1 | 桌台有 `using/idle` 等状态，下单/结账/取消订单时**未见与订单联动** | 开台置 `using`、清台回 `idle`（或 `cleaning` 再人工确认） |
| B2 | 同一桌**多笔未完成订单**的规则未在模型/API 层明确 | 定义「一桌一单」或「一桌多单」策略，并在列表/点餐台体现 |
| B3 | `qr_code` 已有字段，**扫码点餐闭环**（生成链接、顾客端）未接 | 按阶段：先静态桌码 URL，再小程序/H5 |

### C. 支付与对账（能力缺口）

| ID | 现状 / 风险 | 建议方向 |
|----|-------------|----------|
| C1 | 状态机支持 `paid`，但**无支付回调/支付单**域模型 | 引入支付流水表、渠道、回调幂等；再驱动 `TriggerPay` |
| C2 | 无**日结/交班**、无实收与订单汇总报表 | 最小版：按日汇总 + 导出；完整版：班次、操作员维度 |

### D. 厨房与前厅体验

| ID | 现状 / 风险 | 建议方向 |
|----|-------------|----------|
| D1 | 工作台为**拉取列表**，忙时依赖前端轮询 | SSE/WebSocket 推送新单、状态变更；或优化轮询间隔与增量接口 |
| D2 | 订单行无**出菜状态**（整单状态为主） | 按行：`pending/done/served` 或档口维度（可分期） |
| D3 | 无**打印**抽象（后厨单/小票） | 对接云打印或本地代理；先定义打印任务模型 |

### E. 运营与治理

| ID | 现状 / 风险 | 建议方向 |
|----|-------------|----------|
| E1 | 无**沽清**、库存 | 菜单 `sold_out` 或库存扣减（视业态） |
| E2 | 无**操作审计**（谁退单、谁改价） | 关键写操作写审计表或结构化日志 + trace id |
| E3 | 无**备份/迁移/健康检查**运维文档化 | 部署清单：DB 备份、迁移、readiness |

### F. 工程与质量

| ID | 现状 / 风险 | 建议方向 |
|----|-------------|----------|
| F1 | 核心下单/计价/状态机**自动化测试覆盖**可加强 | 单测：状态非法跳转、计价、桌台规则 |
| F2 | `routes.go` 等为 goctl 生成，改 API 需**规范同步** api 定义与前端 SDK | 固定「改接口流程」避免漂移 |

## 四、建议执行顺序（一步一步）

> 以下为**默认偏「先能稳用」**的顺序；若你们不做支付，可将 C 整体后移。

| 步序 | 建议做的块 | 涉及 ID |
|------|------------|---------|
| 1 | 堵住明显业务漏洞：堂食桌校验、服务端计价与总额校验、明细 `menu_id` | A1–A3（A4 可与 1 同期或紧随） |
| 2 | 桌态与订单生命周期绑定 + 一桌一单/多单策略 | B1–B2 |
| 3 | 工作台实时性（至少优化轮询或 SSE） | D1 |
| 4 | 支付流水 + 驱动已支付状态（若需真收款） | C1 |
| 5 | 日结/极简报表 | C2 |
| 6 | 打印或行级出菜（按门店痛点二选一或并行） | D2–D3 |
| 7 | 沽清、审计、运维固化 | E1–E3 |
| 8 | 扫码点餐与营销（视产品路线） | B3 及会员/优惠券等（可另起文档） |

每步完成后：

- 在本文件或「变更日志」中标记完成；
- 若有新发现的缺口，追加到第三节并标注发现日期。
